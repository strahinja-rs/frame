# probe

The perception layer of `/frame`. Surfaces what the user is assuming (without realizing), what they're foregrounding, what they're backgrounding, what's missing entirely. Produces an artifact + probe questions. Question-first: this skill is *built around* asking the right questions, not producing the right structure.

> See [PRINCIPLES.md](../../PRINCIPLES.md) — particularly **§10 question-first**, **§4 Direct-Use Rule** (read the real material), **§11 iteration-as-norm**.

## When to Use

- User signals perceptual uncertainty: "help me see", "figure out", "I don't know what's going on", "something's off".
- Situation as given feels off — vague, internally inconsistent, or implicitly framed in a way that might be cutting off important alternatives.
- High-stakes situation where unsurfaced assumptions would compound expensively.
- Multi-stakeholder situation where the user's framing may have foregrounded one stakeholder at the cost of others.
- Composer-invoked at Phase 1 when signals match "perception first" rule.

## When NOT to Use

- Problem is already well-articulated and the user just wants the canonical structured form → `frame:problem-statement`.
- User is in action mode and probing would be friction → `/approach`.
- Probing has already happened recently with the same context (don't re-probe without new material; iteration is fine but loop-probing isn't).
- The thing the user actually needs is information / research, not perception clarification → `/research` or direct lookup.

## Process

The skill outputs a **Probe artifact**: a structured analysis of the user's current framing, with explicit lists of surfaced assumptions, foreground/background analysis, missing-piece flags, and probe questions ready to ask the user.

Question-first means: the artifact captures the *questions that were generated by reading the situation carefully*, with the *answers (when produced) as derivative output*. The questions are the primary contribution.

### 1. Read the situation carefully (Direct-Use Rule)

If the situation references existing artifacts (a brief, transcripts, prior docs), read them directly per [PRINCIPLES.md §4](../../PRINCIPLES.md#4-direct-use-rule--exercise-the-real-material-not-summaries). Don't probe based on the user's summary of the material — probe the material itself.

If no artifacts, read the situation text/conversation context with care.

### 2. Generate four kinds of probe output

**(a) Surfaced assumptions** — what is the user's framing assuming without saying? Examples:
- "The 6-month timeline is a hard constraint" (assumed but not stated as a hard floor)
- "This is a money decision" (assumed; might be an identity decision)
- "Everyone in scope agrees on what success looks like" (assumed)
- "Past version is the canonical reference" (assumed; might be obsolete)

**(b) Foreground analysis** — what is the user's framing emphasizing? What's at the center of attention?
- "Capacity / 6 months / opportunity cost" foregrounded
- "Relationship and reputation" foregrounded
- "Technical complexity" foregrounded

**(c) Background analysis** — what is the user's framing de-emphasizing or excluding?
- "Family / partner consideration" backgrounded
- "Long-term identity trajectory" backgrounded
- "Other current commitments" backgrounded

**(d) Missing pieces** — what does the framing not include at all that probably matters?
- "No mention of what success looks like to the client (their JTBD)"
- "No mention of who else has done this work for similar clients"
- "No mention of fallback / exit scenarios"

### 3. Generate probe questions

For each surfaced assumption / backgrounded element / missing piece, generate one specific probe question that would help the user clarify their perception. The questions should be:

- **Specific** — "What does done look like to the client?" not "What's the goal?"
- **Inviting reflection, not requiring research** — questions the user can answer from their own knowledge
- **Tight** — one clause, one ask
- **Not leading** — don't bake the answer into the question

Cap at ~5-7 probe questions per probe pass. If more emerge, the situation needs multiple probe rounds (iteration is the norm per [PRINCIPLES.md §11](../../PRINCIPLES.md#11-iteration-is-the-norm-not-the-failure-mode--new-to-frame)).

### 4. Output

Write to:
- **Within composer**: `<contracts-root>/frame-<slug>/probe.md`
- **Standalone**: `<contracts-root>/probe-<slug>.md`

Format:

```markdown
# Probe: <slug>

## Situation as received
<original unstructured situation; verbatim or close>

## Surfaced assumptions
<inferences the user's framing is making without stating them>
- <assumption 1>: <what's implicit>
- <assumption 2>: <what's implicit>
- ...

## Foreground (what the framing emphasizes)
- <element 1>: <how it's emphasized>
- <element 2>: <how it's emphasized>

## Background (what the framing de-emphasizes or excludes)
- <element 1>: <how it's pushed to the edges>
- <element 2>: <how it's pushed to the edges>

## Missing pieces (what the framing doesn't include)
- <missing 1>: <what's absent that probably matters>
- <missing 2>: <what's absent>

## Probe questions (for the user)
1. <specific, inviting reflection>
2. <specific>
3. <specific>
4. <specific>
5. <specific> [up to ~7]

## Recommended next
- <e.g., "Answer probe questions then run frame:problem-statement">
- <e.g., "If multi-stakeholder confirmed, add frame:jtbd before terminator">
- <e.g., "Hand off to /approach with current Polya 9 — frame work is sufficient">

## Extraction-confidence
<low | medium | high — how confident the probe is that surfaced material is real, not invented>

## Assumptions (about the probe itself)
<inferences the framer made about the user's stance, or "none">
```

### 5. Stop

Skill is framing-only. Do NOT answer the probe questions on behalf of the user. The questions are the deliverable — the user (or downstream conversation) answers them.

If invoked from `frame:composer`, return control to composer with the probe artifact. Composer surfaces the probe questions to the user, gathers answers, and decides the next skill.

## Anti-Patterns

- **Answering probe questions on the user's behalf.** Defeats the purpose. The questions are for the user; producing speculative answers in the artifact is hallucination + presumption.
- **Surfacing generic assumptions ("assumes the situation is real").** Vacuous. Surface assumptions that are *specific to this situation* and that the user might not have noticed.
- **Probe questions that are research questions.** "What does the market for X look like?" is research, not perception clarification. Probe questions invite reflection on what the user already knows or feels.
- **Probe questions that are leading.** "Don't you think the 6-month timeline is too tight?" bakes the answer in. "What would have to be true for the 6-month timeline to be tight enough?" invites reflection.
- **Too many probe questions.** Cap is 5-7 per pass. More means the probe wasn't disciplined; iterate in multiple passes.
- **Skipping the Direct-Use Rule.** Probing the user's summary of an artifact rather than the artifact itself misses what the user filtered out — which is exactly what probe is supposed to catch.
- **Restating the situation as the probe.** A probe that just re-summarizes back to the user with no new perception isn't a probe.
- **Question-second.** Producing a structure (table of foreground/background) without the underlying questions being explicit. The questions are the output's logic; without them the structure is decorative.

## Rules

- The skill outputs a single Probe artifact (markdown file); it never executes or answers questions for the user.
- The four output sections (assumptions, foreground, background, missing) are all populated; if any is genuinely empty, mark it as `none` with a one-line note on why.
- Probe questions are capped at ~5-7 per pass (soft cap; can be 8-9 if genuinely warranted).
- When invoked from `frame:composer`, the artifact lives at `<contracts-root>/frame-<slug>/probe.md`.
- When invoked standalone, the artifact lives at `<contracts-root>/probe-<slug>.md`.
- `Extraction-confidence` and `Assumptions` fields are non-optional per PRINCIPLES.md §3.
- Direct-Use Rule (PRINCIPLES.md §4) applies: probe the source material directly when present, not the user's summary of it.
- Question-first (PRINCIPLES.md §10) is mandatory: the artifact's structure follows from the questions, not vice versa.
- When the probe spawns a sub-Agent (e.g., for reading a large artifact in isolation), the contract MUST specify `model: opus` per PRINCIPLES.md §2.
- Recommend the next frame skill in the `Recommended next` section based on what the probe surfaced.
